🔀 Fork 相容性與上游變更影響管理

當 itemhsu（上游）改動時，其他 fork 的 repo 如何不被弄壞 — 設計計劃 · 2026-06-03

狀態：計劃範圍：schema / workflow / 報告格式 / 桌面 App

1. 生態系與所有權（誰會 fork、誰只下載）

itemhsu/tech-rebalance（引擎範本·private）──fork──▶ {user}/tech-rebalance（使用者的引擎，private） itemhsu/tech-rebalance-dashboard（公開 Pages）─fork─▶ {user}/tech-rebalance-dashboard（使用者的儀表板） itemhsu/tradingadmin（桌面 App·public）──不 fork──▶ 使用者「下載 DMG」執行（永遠是 itemhsu 的版本） └─ App 透過 GitHub API 讀「使用者的 fork」
產物使用者怎麼取得變更傳遞特性
引擎程式碼 + schema + workflowfork（複製一份）不會自動更新 → 容易與上游分歧、拿不到修復
桌面 App（GUI）下載 DMG永遠用 itemhsu 最新版 → 但會跟「使用者舊 fork」版本錯位
儀表板前端 html + data.json引擎每日 push 到 dashboard repo同一 fork 內前端與資料每日自我同步
核心風險＝「版本錯位」三種： ① fork 的引擎/schema 舊，但下載的 App 新 → App 期待的欄位 fork 沒有。 ② fork 的 workflow 舊，拿不到上游修復（如本專案的 SendGrid→SMTP、email-log、孤兒 migrator 移除）。 ③ 上游改了 data.json 格式，fork 的舊儀表板前端看不懂。

2. 變更分類 × 衝擊 × 預防

① 策略 JSON / strategy-schema-v3

變更對 fork 衝擊嚴重預防
新增必填欄位fork 既有策略 JSON 驗證失敗 → 當日不執行高 只做加法且選填（帶預設）；新必填欄位走新 schema 版本號，不動舊版
新增選填欄位 / 新 selection method舊策略仍可跑；用不到新功能低validator 容忍未知欄位；method 用 registry 找不到時明確報錯
改欄位語意 / 改預設值同樣 JSON 行為悄悄改變高禁止就地改語意 → 用新欄位名 + 棄用期

② 券商 schema（broker-schema-v2）

變更對 fork 衝擊嚴重預防
schema 加欄位（如本專案的 account_discovery）App/引擎讀新欄位；fork 舊 broker JSON 沒有 → 該功能 fallback中讀取端一律 .get(默認)；$schema 版本號決定 parser
App 新版要求新欄位才能下單fork 舊 broker JSON 無法用新版 App 操作高App 宣告「支援的 broker-schema 版本範圍」；不相容時顯示「請更新券商 schema」

③ 帳戶 schema（accounts.json + Account model）

變更對 fork 衝擊嚴重預防
加欄位（如本專案 strategy_history）舊 accounts.json 沒有 → 用預設；新功能不啟用低已具備：Account model 忽略未知欄位、新欄位 Optional 帶預設
App 寫入新欄位、引擎舊版不讀欄位被忽略（無害）低維持「讀取端寬鬆、寫入端加法」契約
改 Secret 命名（如 _ALPACA_KEY vs _API_KEY）引擎讀不到金鑰 → 交易失敗高命名由 broker schema required_env 驅動；改名走雙寫過渡期 + workflow fallback secrets.A || secrets.B

④ Workflow（.github/workflows）— 最高風險

fork 把 workflow 複製走，之後上游的修復（SendGrid→SMTP、看門狗、孤兒 migrator…）不會自動到 fork。 fork 等於凍結在 fork 當下的版本。
策略說明
瘦 workflowworkflow 只做「checkout → 跑 python run_account.py / migrate_to_mvp.py」，所有邏輯放進版本化的引擎程式碼。如此上游改邏輯時，fork 只要 git pull upstream 一次就拿到，workflow 本身幾乎不用改。
上游同步機制提供 scripts/sync_upstream.sh（git fetch upstream && git merge upstream/main，排除 accounts.json / data/ 等私有檔），或 App「⚙️ 從上游同步引擎」按鈕一鍵執行。
workflow 版本標記workflow 檔頭寫 # workflow_version: N；引擎啟動時比對「程式碼期待版本 vs 實際 workflow 版本」，落後就在 email/issue 提醒「請同步上游 workflow」。
Secrets 別名 fallback所有 secrets.X || secrets.OLD 寫法保留，降低改名衝擊。

⑤ Daily report 格式（data.json + email + dashboard）

變更對 fork 衝擊嚴重預防
data.json 加欄位（本專案 strategy_history / order_alerts）引擎與 email 同 repo → 同步；儀表板前端若舊版可能看不懂中 已具備：data-schema 加法 + 儀表板「data.json 版本不相容」橫幅；引擎每日把對應版本的 mvp_dashboard.html 一起 push → 同 fork 內自我同步
data-schema 改 必填結構舊 data.json / 舊前端壞掉高data-schema-v1 → v2 版本號；前端依 meta.schema_version 切換渲染、不相容顯示橫幅而非崩潰

⑥ 桌面 App（tradingadmin GUI）

變更對 fork 衝擊嚴重預防
App 新版讀新 schema 欄位使用者一律拿新 App，但其 fork schema 可能舊 → 讀不到中App 讀 schema 全程 .get(默認)、防禦式；缺欄位以「—／停用」顯示而非崩潰
App 與引擎的契約改變（如 Secret 命名、accounts 欄位）App 寫的東西引擎不認高App 顯示「相容性檢查」：讀 fork 的 schema_version / workflow_version，落後就提示「請從上游同步引擎」並附一鍵按鈕
App 自身改版所有人下載即得；不分歧低CI 自動發 Release；README 指向 releases/latest

3. 共通原則（防錯設計）

  • 萬物版本化：每個 schema/格式都有 $schema 或 version；讀取端依版本切換 parser、不相容時明確降級或告警，絕不靜默崩潰。
  • 只做加法、欄位選填帶預設；改語意/改必填一律走新版本號 + 棄用期。
  • 讀取端寬鬆、寫入端保守（Postel 法則）：忽略未知欄位、補齊缺欄位預設。
  • 瘦 workflow、邏輯進版本化程式碼：把「會變的東西」集中在可一次 pull 的地方。
  • 可見的相容性告警：版本錯位時，email / 儀表板 / App 都要顯眼提示「請同步上游」，而非默默壞掉。
  • Secrets 別名 fallback：改名永遠保留 新 || 舊。

4. 上游同步機制（讓 fork 拿得到修復）

# scripts/sync_upstream.sh（提供給 fork 使用者；App 可一鍵呼叫）
git remote add upstream https://github.com/itemhsu/tech-rebalance 2>/dev/null || true
git fetch upstream
# 只合併「程式碼/schema/workflow」，保留私有檔（accounts.json、data/、金鑰無涉）
git merge upstream/main -m "sync: pull upstream engine fixes" \
  -X ours -- accounts.json data/ d2p2t6/ weekly_top10/ 2>/dev/null || \
  git merge upstream/main   # 衝突時人工解，但 90% 情況自動
  • App「⚙️ 設定精靈」加一步「從上游同步引擎」，觸發 fork repo 的同步（或開 GitHub「Sync fork」）。
  • 引擎每日 run 比對 code_version vs schema/workflow_version，落後 → email 加「🔄 有上游更新，建議同步」一行。

5. 變更流程與 CI 守門

機制內容
相容性 CI上游每次改 schema → 跑「舊樣本資料 × 新程式碼」與「新樣本 × 舊讀取器」雙向測試（向前 / 向後相容）。本專案已有 Parity / schema 測試可擴充。
棄用政策改名/移除欄位先標 deprecated 並雙寫一段時間（如 1～2 週），email/log 提醒，再移除。本專案 env 命名「雙寫 1 週」即此模式。
migration 腳本需要結構升級時附 scripts/migrate_vN_to_vN1.py，App/同步時自動套用。
版本相容矩陣README/文件維護「App 版本 ↔ 引擎 schema 版本」支援表，App 啟動時對照。
