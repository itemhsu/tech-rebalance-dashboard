3. 逐類衝擊 × 預防（補齊 v1 漏掉的）

選股方法登錄表（#12）— 字串耦合的隱形地雷

策略 JSON 用字串 "method": "top_n_per_group" 指向 selection.py 的函式。 上游改名/移除 method → fork 舊策略找不到 → 整個帳戶當日炸。
  • method 名稱當公開 API：只增不改名；要改走別名表 {舊名: 新函式} + 棄用期。
  • registry 找不到 method 時：明確列出可用清單並 skip 該帳戶（非 traceback）。
  • 測試 CT-SEL-*：枚舉「所有 strategies/*.json 用到的 method」⊆「selection.py 已註冊」。

訂單事件 NDJSON（#9, 16）— 生產者/消費者三方同步

事件型別 ORDER_PLANNED/SUBMITTED/FILLED/REJECTED/STALE 由 trade_log.py 產生、被 trader.reconcile、report_generator.order_alerts、Pages log/index.html 三處消費。改 type 或欄位 → 任一端默默讀不到。

  • event schema 凍結為附加式；新欄位選填。
  • 消費端用 .get()、未知 type 略過不崩。
  • 測試 CT-EVT-*：固定 event 樣本 → 三個消費端都能解析。

Secrets ↔ Workflow 命名契約（#23）

accounts.json.secret_prefix → 引擎找 {PFX}_ALPACA_KEY/_SECRET（alpaca）或 {PFX}_API_KEY/_ACCOUNT_ID（依 required_env）。workflow 的 env: 區塊必須提供同名 Secret。三者一錯就停止交易。

  • 命名由 broker spec auth.required_env 單一來源驅動，不在 workflow 硬寫。
  • 改名保留 secrets.NEW || secrets.OLD fallback（如現有 secret_prefix 仍接受舊 alpaca_secret_prefix）。
  • 測試 CT-SEC-*：每個 enabled 帳戶的 required_env，workflow yml 都有對應 secrets.* 引用。

GUI ↔ 引擎跨倉（#26）

  • App 讀 schema 全程 .get(默認)，缺欄位顯示「—／停用」不崩。
  • App 宣告「支援的 schema 版本範圍」，讀 fork 的 version 落差 → 顯示相容性警告 + 一鍵同步上游。
  • 測試 CT-GUI-*（住 tradingadmin repo，引用共享 fixture）：App 解析「最小合法 accounts/broker/strategy」不丟例外。

4. 共通原則（防錯設計）

  • 萬物版本化：每 schema/格式帶 $id/version；讀取端依版本切換或明確降級告警，絕不靜默崩潰。
  • 只做加法、欄位選填帶預設；改語意/改必填/移除 → 一律新版本號 + 棄用期。
  • 讀取端寬鬆、寫入端保守（Postel）：忽略未知欄位、補缺欄位預設。B 類資料檔尤其要補。
  • 字串耦合點當公開 API：selection method 名、event type、env 名、Secrets 名只增不改名。
  • 瘦 workflow、邏輯進版本化程式碼：把會變的東西集中在一次 pull 可得之處。
  • 可見的相容性告警：版本錯位時 email/儀表板/App 都顯眼提示，而非默默壞掉。

for f in schemas/*.json strategies/strategy-schema-*.json brokers/broker-schema-*.json; do
