🖥 管理者控制台 本機桌面 App

一個跨平台桌面 GUI，集中管理交易系統的所有「人工設定」： 帳戶 CRUD、券商/策略指派、GitHub Secrets、cron 排程、email 收件人等。 不跑 HTTP server、純本機執行、直接操作 repo 與 gh。

📋 執行摘要

一頁看懂

痛點：目前所有管理動作都要手改檔案 + 記指令 — 改 accounts.json、gh secret set、編輯 workflow 的 cron、git commit/push。 容易打錯、漏步驟（例如忘了設對應的 Secret）、也難一眼看全狀態。

解法：一個本機桌面 App，把這些動作收進有驗證、有防呆的 GUI。 它不是新的後端服務 —— 沒有 server、沒有資料庫，直接讀寫 repo 內的 accounts.json / workflow YAML，並 shell out 到 gh 設 Secret。

⚙️ 約束與技術選型

「無 HTTP server + 跨平台 + 本機」如何收斂選擇

關鍵約束「不依賴 HTTP server」直接排除所有 web 方案 （Flask/FastAPI + 瀏覽器、localhost dashboard）。剩下要在「原生桌面 GUI」裡選。

方案無 HTTP?跨平台與本專案契合評語
Python + PySide6 (Qt)✅ 原生✅ Mac/Win/Linux⭐⭐⭐⭐⭐推薦。專案本身 Python，GUI 可直接 import accounts 驗證/broker spec，零跨語言成本；Qt 介面專業
Python + Tkinter✅ 原生✅（內建）⭐⭐⭐⭐零額外依賴（Python 自帶）。介面陽春，但最穩、打包最小 → 備案
Tauri (Rust+webview)⚠️ IPC 非 HTTP✅⭐⭐二進位小、UI 漂亮，但要寫 Rust + 前端，跨語言橋接本專案邏輯，過重
Electron⚠️ IPC 非 HTTP✅⭐⭐技術上不需 HTTP server，但體積大、又是 JS 生態，與 Python 後端要 IPC 橋接
Web + 瀏覽器❌ 需 server—✗違反「不依賴 HTTP server」，出局
選定：Python + PySide6（Qt for Python） 無 HTTP/socket、跨平台、與既有 Python 系統同生態（可直接重用 brokers/registry.py 驗證 broker、strategies/*.json 列策略、 engine/universe_loader.py 等）。打包用 PyInstaller 出三平台單檔。
備案：若要求「零安裝相依」，退回 Tkinter（Python 內建）。

🔒 安全模型（金鑰）

最關鍵：金鑰絕不落地
  • App 永不持久化任何 secret。設 Secret 時，密碼欄輸入 → 直接以 stdin 餵給 gh secret set NAME --body - → 進 GitHub Secrets，不寫 accounts.json、不寫任何本機檔、不寫 log。
  • Secrets 分頁只顯示「存在 / 不存在」（來自 gh secret list），從不顯示值。
  • 輸入框遮罩、剪貼簿不留存、記憶體用完即清。
  • 每個帳戶依 secret_prefix + broker 算出「應有哪些 Secret」，缺的標紅 → 防「新增帳戶卻忘了設金鑰」這類漏步驟。
  • git push 含金鑰風險防護：commit 前掃描 diff，若疑似 token 字串出現在被 commit 的檔案 → 擋下並警告。
界線：App 幫你「呼叫 gh 設 Secret」， 但 gh 的登入授權仍由你本人完成（gh auth login）。 App 不碰你的 GitHub 帳密，只用你已授權的 gh session。

#現況問題需求
1secret_prefix 要使用者填自動預設 ACC{id}，表單不顯示（進階模式才可改）
2data_dir 使用者不懂是什麼自動產生 data/{id}，表單不顯示
3use_new_runner 是工程概念從表單移除，內部固定 true
4「啟用」語意不清改成明確說明：「啟用 ✅ = 每日自動交易；關閉 = 暫停這個帳戶不下單」
5相關 Secret 要切到別頁設，使用者不會知道表單內就能填 API Key / Secret（依券商動態欄位），存檔時一併寫入 GitHub Secrets，免切分頁
6存檔後不知道設對沒存檔前用填入的金鑰試打券商 API（如 /account）確認連線；成功才允許存檔離開，失敗顯示原因（如 live key 配 paper）
7使用者要填 idid 由系統自動指定（取下一個可用號），使用者不填、不顯示為輸入欄
8沒有人看得懂的識別新增必填「帳戶名稱」欄（對應 label），讓使用者填識別文字（如「我的科技股」）——這是唯一需要使用者命名的欄位
9獨立 Secrets 分頁多餘金鑰改為帳戶表單內嵌（#5）後，Secrets 分頁移除；Secret 狀態折進帳戶列的「Secret 燈」，缺金鑰在帳戶表單就提示 → 分頁由 5 變 4（總覽 / 帳戶 / 排程 / 設定）
10無稽核軌跡建立操作日誌：記錄所有帳戶管理動作（新增 / 編輯 / 刪除 / 啟用停用 / 設金鑰 / 改 cron），含時間戳、動作、對象帳戶、結果；不含金鑰值。存本機 log 檔，總覽分頁顯示最近數筆
11總覽與帳戶分頁重複（都列帳戶 NAV）總覽重新定位為「全域管理」——只放跟個別帳戶無關的設定（參考 setup-guide）：Email 發送、Dashboard 推送、環境/登入狀態、操作日誌。帳戶 NAV 只留在「帳戶」分頁。原「設定」分頁與新總覽重疊 → 合併進總覽，分頁由 4 變 3（總覽 / 帳戶 / 排程）
12填了 mail 金鑰不知道能不能發總覽設 Email 後必須「測試發信」成功才放行——實際寄一封測試信，SMTP 回成功（可選：問「收到了嗎」）才允許離開；失敗顯示原因（如 App 密碼錯）。與帳戶金鑰「測試連線才存」同模式
13gh 沒登入會整串壞掉gh 主動驗證（非只顯示狀態）：未登入 / 無權限 → 擋下相關動作並給 gh auth login 指引，確認可用才放行
14新使用者不知怎麼開始首次啟動精靈含「fork 助手」：幫使用者把這個 public 專案 fork 到自己 GitHub（gh repo fork）→ clone → 引導設全域 Secret → 啟用 Actions。讓非工程者也能擁有自己的可運行實例
15看不到系統發生什麼事全域 log 讀取（總覽）：可檢視 ① 操作日誌 audit.jsonl（誰改了什麼）② 每日 cron 執行歷史（gh run list 成功/失敗）。可捲動、依動作/結果篩選、開完整內容
16不知道某帳戶實際下了什麼單帳戶 log 讀取（帳戶分頁選一列 → 檢視該帳戶）：交易事件 data/{id}/trade_events.jsonl（下單/成交/換股）+ NAV 歷史 portfolio_state_history.json。唯讀、時間倒序
17EMAIL_SENDER 被當機密遮罩寄件人 email 不具機密性 → 顯示實際值（明文、可編輯），不放遮罩 Secrets 清單。存本機 admin_gui/data/config.json；只有 App 密碼是 Secret
18SendGrid 已棄用仍出現移除 SendGrid：GUI 全域 Secret 清單、probes、（後續）production email fallback 都拿掉 SENDGRID_API_KEY
19DASHBOARD_PUSH_TOKEN 可否移除？現行模型（推到獨立 dashboard repo）下不可。但見需求 21：改部署模型即可徹底移除
20測試發信要使用者填一堆測試發信「給自己」：收件人自動 = 寄件人、無收件人欄；只需輸入一次 App 密碼（Secret 讀不回，故設定即測），驗證成功才把密碼寫進 GitHub Secrets
21DASHBOARD_PUSH_TOKEN 要使用者建 PAT、麻煩又易錯不要使用者填 token。共用「預設 token」不安全（等於大家共用一把，能推別人的 repo）→ 正解：dashboard 改從 fork 自己的 GitHub Pages 發佈，用 Actions 內建 GITHUB_TOKEN（自動、零設定）。不需 PAT、不需獨立 dashboard repo，使用者一個字都不用填
22測試發信還要再輸入 App 密碼測試發信不要密碼、只顯示結果（正常/失敗）。原因：本機 GUI 拿不到 Secret 值（gh 讀不回），要在本機寄就得再輸入密碼。正解：不在本機寄，改觸發雲端 test_email.yml workflow（它有 EMAIL_PASSWORD secret）→ GUI 讀 run 結果顯示「✅ 正常 / ❌ 失敗」。使用者零輸入（前提：EMAIL_PASSWORD 已設，否則提示先設）
