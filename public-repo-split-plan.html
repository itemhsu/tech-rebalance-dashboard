4. 廢棄 tech-rebalance-data

  • 為何：它是 P3 用 vendored wheel 臨時建的重複 Repo B；角色已被 tech-rebalance（private）涵蓋。
  • 做法：① GitHub → Settings → Archive（封存唯讀）或先停用其 Actions；② 確認 private 薄殼跑通後，由你親手刪除（永久刪資料的動作我不代做）。
  • 它的 paper 對帳價值已達成（驗證了「裝引擎 → 跑 → 下單」整條鏈路），可功成身退。

5. 退役鏡像/安全掃描機制

v1 的 build_pub_tree.py + pub_allowlist.txt + 安全掃描是「私有→pub 過濾鏡像」用的。 既然引擎直接住 pub、在 pub 開發，就不再需要持續鏡像。

  • 它們已完成一次性任務：把乾淨的初始引擎碼種到 pub。
  • 保留 build_pub_tree.py 當「一次性遷移工具」歷史紀錄即可；不接自動化、不再跑。
  • （若日後又想私有開發+鏡像，再啟用——但目前模型不需要。）

8. 風險與原則

  • private 是真錢系統：T2 並行 dry-run 對帳通過才切（T3）；舊 workflow 切換期間保留可回退。
  • 引擎來源移轉：T1 起只在 pub 改引擎；private 引擎碼凍結，避免雙頭開發分歧。
  • 廢棄 = 先封存後刪：tech-rebalance-data 先 Archive，確認新薄殼跑穩，硬刪由你親手。
  • 金鑰全程你親手：private 的 Secrets 保留不動；切換不碰金鑰值。
一句話：tech-rebalance（private）本來就是 Repo B；把引擎搬到公開的 tech-rebalance-pub、私有改成「裝 pub 引擎來跑」， tech-rebalance-data 因角色重複而廢棄，鏡像機制退役。核心是 T2 並行對帳——真錢系統先並存、對帳一致、再切換。
