1. 目標與動機

現在引擎是 private，Repo B 要 pip install 它得放一個 PAT（ENGINE_INSTALL_TOKEN），或用 vendored wheel（要手動換檔）。兩者都麻煩。

新做法：不動現有 private repo（它有金鑰、live 帳戶、資料，繼續跑）。 另開一個純碼的 itemhsu/tech-rebalance-pub（public），把「需要公開的引擎碼」鏡像過去。 Repo B 直接 pip install git+https://github.com/itemhsu/tech-rebalance-pub@v1.0.6 ——免 token、免 vendored wheel。

4. 同步機制（private → pub，單向鏡像）

private repo 新增 .github/workflows/publish_pub.yml，在 v* tag 觸發：

1. checkout private
2. 依「白名單」把允許的路徑複製到乾淨的 /pub_tree
3. ★ 安全掃描：/pub_tree 內不得出現任何禁列檔（accounts.json、data/*/portfolio_state、
   .env、live workflow…）→ 命中即 fail，絕不推送
4. 用 PUB_PUSH_TOKEN（fine-grained PAT，只給 tech-rebalance-pub 寫權限）
   force-push /pub_tree 到 tech-rebalance-pub，並打同名 tag
5. pub 的 release.yml（被 tag 觸發）建 wheel + manifest + Release
白名單以一份 scripts/pub_allowlist.txt 為單一事實來源；複製與掃描都讀它，避免兩處不一致。

5. 安全防線 ★（零個資外洩）

  • 白名單為主：只複製明列路徑；新增檔案預設不進 pub（漏列只會少東西，不會外洩）。
  • 推送前掃描：/pub_tree 比對禁列 glob（accounts.json、**/portfolio_state*.json、**/trade_events.jsonl、.env、**/data/[0-9]*/**、live workflow 名）→ 命中即中止。
  • 內容掃描：再 grep 一次疑似金鑰/email（ALPACA、@gmail、PAT 樣式）→ 命中即中止。
  • pub 端 CI 守門：pub repo 自己有測試 CT-PUB-CLEAN，每次 push 掃自己有沒有禁列檔（雙保險，萬一鏡像出錯也擋）。
  • Token 範圍最小：PUB_PUSH_TOKEN 只給 tech-rebalance-pub 的 Contents:write，不碰 private repo、不碰其他。
  • pub 設 branch protection：唯讀鏡像，禁止人工直接 push（只接受 publish workflow）。

8. 測試矩陣 ★（完整）

U單元 I整合 CCI/E2E。安全類為最高優先。

A. 白名單 / 建樹（build_pub_tree）

UPT-01 copies_allowlisted白名單內的檔（runner.py/engine/…）都被複製進 pub_tree
UPT-02 data_fileleveldata/ 只複製 universe.json+shares+__init__；data/1、data/3 不複製
UPT-03 excludes_unlisted未列檔（accounts.json、live workflow）不在 pub_tree
UPT-04 allowlist_single_source複製與掃描讀同一份 pub_allowlist.txt

B. 安全掃描（★最高優先）

USEC-01 reject_accounts_jsonpub_tree 含 accounts.json → 掃描 fail（中止推送）
USEC-02 reject_state_files含 **/portfolio_state*.json / trade_events.jsonl → fail
USEC-03 reject_account_dirs含 data/3/...、d2p2t6/data/... → fail
USEC-04 reject_env_and_live_wf含 .env 或 daily_all_accounts.yml → fail
USEC-05 content_scan_keys內容出現 ALPACA 金鑰樣式 / @gmail / PAT 樣式 → fail
USEC-06 clean_tree_passes正確的白名單樹 → 掃描通過

C. 鏡像同步 / 發版（publish_pub）

IPUB-01 push_argspush 目標是 tech-rebalance-pub、用 PUB_PUSH_TOKEN、打對的 tag（mock git/gh）
IPUB-02 abort_on_scan_fail掃描失敗 → 不執行 push（先掃再推）
CPUB-03 pub_release_assetstag 後 pub Release 含 wheel + manifest（沿用 release.yml 驗證）
CCT-PUB-CLEANpub repo 自身 CI：掃自己無禁列檔（雙保險）

D. 安裝 / 端到端

CINST-01 pip_from_pub_no_token乾淨 venv pip install git+…pub@vX（無 token）→ run-account --help 可跑、資產定位
CINST-02 repob_dryruntech-rebalance-data 改 git+ 安裝後 dry-run 綠燈、選股與舊一致

E. GUI 指向 pub

UGUI-01 services_target_pubengine_release/manifest/provisioner 的引擎 repo 常數＝tech-rebalance-pub
UGUI-02 no_token_pathprovisioner 產的 daily.yml 用 git+ 安裝、不含 ENGINE_INSTALL_TOKEN
GGUI-03 headlessGUI 全套件 offscreen 無回歸
一句話：不動 private（live 不停），另開純碼 public 鏡像，用白名單複製 + 雙重安全掃描 + pub 端 CI 守門確保零個資外洩； Repo B 與 GUI 從此免 token、免 vendored wheel。最關鍵防線是 §5 與測試 B 類（安全掃描）——寧可少推檔，不可外洩一個位元組。
