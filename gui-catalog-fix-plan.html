三、三種 policy（檔案處理規則）

policy缺檔時已存在時用途
render從 templates/ 產生（代入版本）覆蓋（版本要更新）daily.yml · test_email.yml
placeholder建空檔/佔位不動data/.gitkeep · dashboard 首檔
protected不建不碰accounts.json · data/{id}/
policy 是 manifest 的欄位（metadata），不是實體目錄。 檔案實體位置由功能決定（workflow 在 .github/workflows/、accounts.json 在根）。

GUI 修復 = 讀 manifest，套 policy（一個通用迴圈，邏輯固定）

def sync(repo, section, manifest, version):
    for e in manifest[section]:
        p = e["policy"]
        if p == "render":
            body = fetch_pub(e["src"]).replace("{version}", version)
            put(repo, e["path"], body)              # 覆蓋
        elif p == "placeholder":
            if not exists(repo, e["path"]):
                put(repo, e["path"], _seed(e["path"]))   # 缺才建
        elif p == "protected":
            pass                                    # 永不碰
        # 未知 policy → 跳過（安全預設，不當機）
新增檔案 = 在 repo_template.json 加一行（+ render 的話在 templates/ 放來源）。GUI 程式碼一行不改。 建立（build）與修復（repair）共用這個 sync —— 建立=全套、修復=補缺。

六、三層驗證（守門，防止填錯/漏填）

第一層：Schema（結構正確）

schemas/repo-template-schema-v1.json（draft-07）擋掉：
  • 未知 policy（typo）— policy 限 enum [render, placeholder, protected]
  • render 沒填 src — if policy==render then required:[src]
  • placeholder/protected 卻填 src — then: not required:[src]
  • 多餘欄位 — additionalProperties:false

第二層：一致性檢查（schema 擋不到的）

scripts/validate_repo_template.py：
檢查防止
V4 同 section path 不重複同檔填兩次
V5 每個 render 的 src 在 templates/ 真存在指向不存在的範本
V6 templates/ 每個檔都被引用（無孤兒）加範本卻忘了登記（核心防護）
V7 path 無 ../、不以 / 開頭路徑逃逸

第三層：CI 自動跑 + GUI 防禦

  • pub engine CI：validate_template.yml，push 動到 repo_template.json / templates/ / schema 就跑 validator，紅燈不能 merge
  • GUI 防禦：抓 manifest 後先用 schema 驗；缺檔或不合 schema → 退回 GUI 內建 fallback（硬編 4 必要檔）+ 警告，App 不當機

七、綠燈卡死的解法

根因：_set_done(done=True) 會 disable 按鈕。舊版用戶 repo「存在但破損」→ 綠燈 ✅ → 按鈕變灰 → 無法修 → 卡死。
解法：repo 存在時「建立」按鈕變「🔧 修復」（仍可按，不變灰），點擊跑同一個 sync()（冪等）。 render 覆蓋（更新版本+修 owner）、placeholder 補缺、protected 永不碰用戶資料。可安全重複按。
