🧩 JSON-Driven 券商引擎 新券商 0 程式碼

把「每家券商一個 Python client」升級為「一個通用 REST 引擎 + 每家一份 JSON」。 新增 REST 券商 = 寫一個 spec JSON，0 行 Python，與策略層相同的 SOP。

📋 執行摘要

一頁看懂這份計劃

現況：策略層已 JSON-driven（新策略 0 程式），但券商層還是「每家一個 client」 （brokers/alpaca_client.py ~250 行）。要加 Tradier 就得再寫一個 client。

目標：把券商層也 JSON 化。寫一次通用 REST 引擎 brokers/rest_broker.py，之後每家券商只寫一份描述「API 長怎樣」的 JSON （auth 方式、endpoint、請求/回應欄位對應），不寫 Python。

軸新增一個要做什麼JSON 化後
策略寫 v3 JSON✅ 已是 0 程式
券商（本計劃）寫一個 client.py→ 寫一份 spec JSON，0 程式
⚖️ 誠實的取捨 通用引擎(~300 行) + schema 擴充是一次性高成本，比單寫一個 Tradier client(~150 行)還多碼。 兩家券商時其實不划算，從第 3 家 REST 券商起才回本。本計劃之所以做，是因為「券商也 JSON 化」是明確的設計要求，且回本後永久受惠。

🤔 為什麼券商「也能」JSON 化

差異是行為，但行為可被宣告式描述

券商間的差異看似是「行為」（form-encoded vs JSON、欄位名、巢狀回應）， 但這些行為都是有限、可列舉的模式 —— 只要引擎支援這些模式，差異就退化成「填哪個值」的資料問題。

差異點AlpacaTradierJSON 怎麼描述
請求編碼JSONformrequest.encoding: "json"｜"form"
認證2 個 headerBearer tokenauth.method + header_template
下單欄位名time_in_force / qtyduration / quantityrequest.field_map
固定參數無class=equityrequest.constants
路徑帳號無/accounts/{id}/…endpoints 模板
回應取 ididorder.id（巢狀）response.order_id_path
餘額欄位portfolio_valuebalances.total_equityresponse.balance.*
TIF 值翻譯dayday（但 gtc→gtc）value_maps.time_in_force
界線 能 JSON 化的前提是差異「可列舉」。一旦某券商需要自訂邏輯（簽章演算法、OAuth refresh 流程、本地 gateway 握手）， 純 JSON 表達不了 —— 那種券商仍需特例程式（見 涵蓋範圍與限制）。

🎯 範圍與不做什麼

避免 scope creep

✅ 做

  • broker-schema v1 → v2：加 request / response / value_maps 區塊
  • 寫通用 brokers/rest_broker.py（RestBrokerClient，實作 BrokerClient ABC）
  • Alpaca 改成純 JSON（補對應表）+ parity 測試證明 live 帳戶 0 變動
  • Tradier 純 JSON + sandbox 驗證
  • 保留 BrokerClient ABC 不變（外部呼叫者 runner.py 完全不動）

❌ 不做

  • 不碰策略層 / selection.py / runner.py（介面不變）
  • 不支援需自訂邏輯的券商（OAuth refresh、本地 gateway）—— 留特例 hook
  • 引擎開發階段（A–D）只在 sandbox 驗證 —— 蓋水管不用真錢測漏。live 不是被排除，是放在 Phase E 過安全閘後上線（見下方）
  • 不重寫既有測試框架，沿用 pytest + mock

📐 broker-schema v2

新增 3 個宣告式區塊

v2 在 v1 之上新增 request / response / value_maps。 v1 spec（只有 client_class、無對應表）仍可被舊 client 路徑載入 —— 向下相容。

// tradier.json（v2 新增區塊）
{
  "request": {
    "encoding": "form",              // "json" | "form"
    "field_map": {                    // 通用名 → 券商欄位名
      "symbol": "symbol",
      "qty": "quantity",
      "side": "side",
      "order_type": "type",
      "time_in_force": "duration"
    },
    "constants": { "class": "equity" }   // 每單固定帶
  },
  "response": {
    "order_id_path": "order.id",        // 巢狀路徑
    "order_status_path": "order.status",
    "balance": {
      "nav":  "balances.total_equity",
      "cash": "balances.total_cash",
      "buying_power": "balances.total_cash"
    },
    "positions_path": "positions.position",
    "quote_list_path": "quotes.quote",
    "quote_symbol_field": "symbol",
    "quote_price_field": "last"
  },
  "value_maps": {
    "side": { "buy": "buy", "sell": "sell" },
    "time_in_force": { "day": "day", "gtc": "gtc" }
  }
}
路徑語法 order.id 走點號逐層；list 取用 positions.position[]。 引擎用一個小 helper _dig(data, "a.b.c") 解析，不引入 JSONPath 套件（減依賴）。 Tradier 的「單筆回 dict、多筆回 list」由引擎統一正規化成 list。

⚙️ 通用 REST 引擎

brokers/rest_broker.py — 寫一次

RestBrokerClient(BrokerClient) 讀 v2 spec，把 ABC 的 7 個方法用「查表 + 組請求 + 解析回應」實作。

ABC 方法引擎怎麼做
get_account_balanceGET endpoints.balances（套 account_id）→ 用 response.balance.* 取值
get_positionsGET positions → 走 response.positions_path → 正規化 list → 欄位映射
get_latest_pricesGET market_data.quotes_endpoint → quote_list_path + price field
place_orderfield_map + constants + value_maps 組 body → 依 encoding 送 json/form → order_id_path 取 id
is_trading_dayGET clock/calendar → 依 spec 的 market_data.* 解析 open/closed
cancel_all_open_orders列 open orders → 逐筆 DELETE（路徑套 account_id + order_id）
wait_for_fills輪詢 order_by_id → 用 order_status_path 判 terminal 狀態

共用設施沿用既有：_request（retry/backoff）、_check_resp（401→BrokerAuthError、不洩 secret）、 check_capability（fractional 擋下）。引擎只多了「查表組裝」那層。

特例 hook（給不純 REST 的券商留後路） spec 可選填 integration.client_class 指向自訂子類；有填就用自訂，沒填就用通用引擎。 Alpaca/Tradier 走通用引擎；未來 IBKR/Schwab 真要做時用 hook 寫特例，不破壞 JSON-first 主路徑。

🗺 對應表範例：同一筆下單 → 兩家

證明同一通用呼叫被引擎翻成兩種 HTTP
# 通用呼叫（runner.py 不知道是哪家券商）
client.place_order("AAPL", qty=10, side="buy",
                   order_type="market", time_in_force="day")
Alpaca（encoding=json）Tradier（encoding=form）
URLPOST /v2/ordersPOST /accounts/VA…/orders
Body {"symbol":"AAPL","qty":"10","side":"buy","type":"market","time_in_force":"day"} class=equity&symbol=AAPL&quantity=10&side=buy&type=market&duration=day
取 idresp["id"]resp["order"]["id"]

兩種輸出完全由各自的 JSON 對應表決定，引擎程式碼一行都不為某家特判。

🧱 涵蓋範圍與限制

誠實標出 JSON 化的邊界
券商類型例純 JSON?說明
REST + 永久 token/keyTradier, Alpaca, Public, Webull✅ 可本引擎目標範圍
REST + OAuth refreshSchwab, tastytrade, TradeStation⚠️ 半API 形狀可 JSON，但 token refresh 流程需特例 hook
本地 gateway / SDKIBKR, moomoo, Shioaji❌ 否需常駐 gateway 握手，非純 HTTP → 走 client_class 特例
不要過度承諾 「新券商 0 程式」只對純 REST + 永久憑證的券商成立。 OAuth/gateway 券商仍需少量特例程式 —— 這是引擎的設計邊界，不是 bug。

ETradier 上線實單（go-live）

⏱️ 安全期後 · 這才是「有用」的終點

前置條件（缺一不可）

  • accounts.json 新增一筆 broker: tradier, environment: live
