1. 問題陳述

每新增一個帳戶/策略，就要手寫一個近百行的產生器函式 —— 並且每次都先壞一次、再修一次。 這次帳戶 #3（mom_6m_t20）就是：報告日期長期停在舊資料、主旨錯成「Monthly TOP10」、三個按鈕消失， 全因為 migrate_to_mvp.py 裡根本沒有 #3 的產生器。

現況：migrate_to_mvp.py 為每個策略硬寫一個函式：

migrate_d2p2t6()       # 帳戶 #2
migrate_mom_6m_t20()   # 帳戶 #3   ← 這次才補上（頭痛醫頭）
migrate_weekly_top10() # 帳戶 #4
migrate_d2p2t6_live()  # 舊
migrate_top9psq_live() # 舊
… 每個 ~90 行，彼此 90% 重複

~400預期還要再錯的次數

2. 根因分析：其實只有「兩個地方」會變

把六個函式逐行比對，95% 完全相同（讀 state、抽 nav_history/trade_log、呼叫 write_data_json、注入 benchmark、存快照）。真正隨策略不同的只有兩點：

變異點各策略目前怎麼做本質
① 持股清單來源 state.top10 / state.target_weights / positions 同一組欄位的 fallback 順序，已可泛化
② 排名（rankings）形狀 ranked_stocks（TOP10/MOM）、scorecard（Weekly）、 三大宇宙分組（D2P2T6） 3 種已知形狀，可用「宣告式 adapter」表達

其餘（路徑、strategy_id、id/label）本來就都在 accounts.json 裡， 根本不該寫死在 Python。

3. 設計原則

  • 新增策略 = 只改 JSON、0 行 Python（與本專案既有的 SpecEngine、 rest_broker「JSON-driven」哲學一致）。
  • 單一產生器 generate_for_account(account)，迴圈跑 accounts.json 的每個 enabled 帳戶。
  • 變異點用宣告式設定表達（在 strategy JSON 的 dashboard 區塊），不是 if/elif 分支。
  • 產出一律經 validate_data_json() 驗證 + 套用 email schema（含 cta），壞掉當場失敗而不是無聲產生爛報告。

4. 提案架構

def generate_all(output_dir):
    for account in load_accounts(enabled=True):          # 來源唯一：accounts.json
        generate_for_account(account, output_dir)

def generate_for_account(account, output_dir):
    strat  = load_and_validate(account["strategy"])      # 策略 JSON（含 dashboard 設定）
    state  = load_state(account["data_dir"])             # 統一 portfolio_state.json
    if not state: return "skip"                          # 無資料 → 略過（不算錯）
    hist   = extract_history(account["data_dir"])

    holdings = resolve_holdings(state)                   # ① 泛化 fallback
    rankings = resolve_rankings(state, strat)            # ② 宣告式 adapter（見下）

    data = write_data_json(strat, account, state, hist, holdings, rankings, ...)
    inject_benchmark(data); validate_data_json(data); save_snapshot(data)
    return "ok"

↑ 不論 1 個或 400 個帳戶，都是這段、沒有任何 per-strategy 分支。

① resolve_holdings（持股 fallback 鏈）

state.get("top10")
  or list(state.get("target_weights", {}).keys())
  or [p["symbol"] for p in state.get("positions", [])][:N]

② resolve_rankings（宣告式 adapter）

在 strategy JSON 宣告排名來源，產生器照表操作；新策略若用既有形狀，一個字都不用寫：

dashboard.rankings.source讀取適用
"ranked_stocks"（預設）state.ranked_stocksTOP10 / MOM 系列（多數）
"scorecard"state.scorecard（score→rank）Weekly TOP10
"universe_groups"依 dashboard.groups 設定分組D2P2T6（軍火/醫藥/科技）
沒設 rankings.source 時，產生器自動偵測 state 內有哪個欄位 （ranked_stocks → scorecard → 由 positions 推導）， 所以絕大多數策略連這行設定都不用加。

5. 遷移計畫（對真錢報告安全）

階段動作安全保證

P2Parity 測試：對 #1/#2/#4 跑「新產生器 vs 舊函式」，逐欄比對 data.json 一致輸出位元級相同才算過
P3workflow 改呼叫 generate_all；觀察 1 天（與舊輸出並存比對）可隨時回退
P4刪除 6 個 migrate_* 函式（含這次的 migrate_mom_6m_t20）移除 ~500 行重複碼

6. 測試計畫

ID測試嚴重度
R-01resolve_holdings：top10 / target_weights / positions 三種來源各取對高
R-02resolve_rankings：ranked_stocks / scorecard / universe_groups / 自動偵測 各產出正確形狀高
R-03Parity：#1/#2/#4 新舊輸出逐欄一致（防回歸）高
R-04對 accounts.json 每個 enabled 帳戶跑產生器 → validate_data_json 全過、date=最新、subject 可套版、含 cta高
R-05新增一個「假策略 JSON」→ 不寫任何 Python，產生器即能產出有效 data.json（證明 0-Python 目標）中
R-06無資料帳戶 → 回 skip、不丟例外、不產生爛報告中

這次已先加的守門測試（email section 名有效、subject 可套版、含 cta、migrate 有對應產生器）併入 R-04 範圍。

7. 對比（重構前 / 後）

現在（每策略硬寫）重構後（資料驅動）
新增 1 個策略寫 ~90 行 Python + 接 main + 容易漏 → 報告壞只放 strategy JSON（多數連 rankings 設定都不用）
產生器程式碼量6×~90 ≈ 540 行，且隨策略線性增長~1 個泛用函式 + 2 個 resolver（固定）
新策略漏接的後果無聲產生舊/錯報告（如 #3）accounts.json 一列就自動納入；缺資料則 skip
22→400 策略~400 次「錯一次、修一次」零增量 Python

8. 風險與緩解

  • D2P2T6 分組排名是唯一較特殊的形狀 → 用 universe_groups adapter + 策略既有的群組設定表達，Parity 測試把關。
  • 真錢 #3：本重構只動「報告產生」，不碰交易/下單；P2 Parity + P3 並存比對確保輸出不變。
  • 舊函式分階段刪除，任何階段可回退。
結論：把 migrate_to_mvp.py 從「每策略一函式」改成「讀 accounts.json + strategy JSON 的單一資料驅動產生器」， 讓「新增策略 = 只改 JSON、0 Python」。這才是根治，而非再補第 7、第 8… 個 migrate_*。
