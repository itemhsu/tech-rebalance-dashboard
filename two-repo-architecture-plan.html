引擎碼(共享) vs 使用者系統(設定+資料+執行) — 用「版本引用」取代「fork 複製」 · 2026-06-03

狀態：設計提案全程雲端，無本機運算自託管金鑰不變
1 診斷2 目標架構圖3 兩條分歧軸×三層散佈 4 工作目錄解耦 ★5 遷移分階段6 compat→release gate7 風險與決策

1. 診斷：錯在「用 fork 散佈需要中央演進的引擎碼」

fork 是整包複製。但 repo 裡混了生命週期完全不同的東西，其中只有「引擎碼」是需要中央更新的——它卻被複製進每個 fork。 我們做的 sync_upstream / drift 偵測 / 跨版 compat 測試，全是在補救「把該共享的東西複製掉」這個決定。
類別範例該怎樣現在
共享引擎碼runner.py、engine/、portfolio.py、schemas/、strategies/、workflow中央更新、版本引用❌ 被複製進每個 fork
個人設定accounts.json、Secrets每人一份✅ 對
個人資料data/{id}/ state/history/events每人一份✅ 對
執行GitHub Actions(自託管金鑰)每人自己跑✅ 對

3. 兩條分歧軸 × 三層散佈

fork 會分歧的有兩種：①引擎碼 ②編排(workflow)。三種散佈機制分屬不同層，可疊加。

機制解哪條軸跑在哪更新動作取捨
① pip 套件碼Repo B 的 Actions 雲端@v1.3 → @v1.4最輕、Python 原生；每跑裝相依(~30s，可快取)
② 容器碼+環境同上雲端:1.3 → :1.4環境完全鎖死；多維護 Dockerfile。啟動~1s，拉映像~15–40s（日批次無感）
③ 可重用 workflow編排同上雲端@v1 → @v2連 workflow 都不分歧；內部仍用①或②取得碼
疊加最終長相（兩條軸一次鎖住）： 
Repo B workflow（3 行）
  └─ uses: itemhsu/tech-rebalance/.github/workflows/daily.yml@v1   ← ③ 編排中央化
        └─ pip install tech-rebalance@v1.3                        ← ① 碼中央化
              └─ run-account --all
建議順序：先 ①（地基）→ 疊 ③（消滅 workflow 分歧）→ ② 當備案（遇難裝原生相依再上）。

4. 成敗關鍵：引擎「工作目錄解耦」★

現在引擎到處用 Path(__file__).resolve().parent[.parent] 假設「碼和資料同 repo」。要分家，引擎必須區分兩種根： package_root()（引擎自帶資產，隨套件走）vs workdir()（使用者設定+資料，注入）。

新增 engine/paths.py

import os, pathlib
def package_root():   # 引擎自帶：strategies/ schemas/ brokers/ 參考資料
    return pathlib.Path(__file__).resolve().parent.parent
def workdir():        # 使用者：accounts.json + data/（預設 cwd，可用 TR_WORKDIR 覆寫）
    return pathlib.Path(os.environ.get("TR_WORKDIR", os.getcwd()))

6. compat 測試 → 引擎套件 Release Gate（沒白做）

這套相容性測試從「跨 fork 救火」升級成「引擎套件發版前的品質閘」——這才是它們本來最該待的位置。

測試現在角色新角色（Repo A 發版 gate）
CT-SCHEMA / unit1·3 / brokers/schemafork 內驗證發版前驗證所有內建 schema 合法
CT-VER-PRESENTfork lint套件 build 前 lint（$id/version 齊全）
CT-BREAK-DETECT跨 fork 防破壞發版時比對上一個 release tag 的 schema → 決定 semver（破壞=major bump，擋誤發）
CT-BACK-CORPUS（golden）跨版回放每次發版存一份 golden；新版套件回放歷年樣本 → 保證讀得了舊使用者資料
CT-FWD-TOLERANT載入器寬鬆套件契約測試（吃未來欄位不崩）
CT-ALIGN-VERdata.json↔前端套件 SCHEMA_VERSION ↔ 前端支援 對齊 gate
CT-ALIGN-SEC / RUNfork workflow 對齊移到 Repo B 模板的 lint（薄 workflow 金鑰注入正確）
CT-SEL / CT-EVT字串耦合套件契約測試
CT-MIGRATE-RT生成 idempotent套件契約測試
CT-GUI-CONTRACTApp↔forkApp↔引擎套件版本相容
B 類資料檔 schemafork 資料驗證隨引擎走；Repo B 資料的版本依據
sync_upstream / drift / email 落後提示fork 救火★ 退役（版本 pin 取代「拉取分歧修復」）
新增的發版 gate：CT-PKG-INSTALL（裝好套件後 run-account --dry-run 在乾淨目錄可跑）、 CT-PKG-PARITY（新版套件對 golden 樣本輸出與上一版一致，除預期改動）。

7. 風險與要先決定的點

決策建議理由
引擎 repo 公開/私有公開純碼無秘密；Repo B pip 安裝免 token（私有要放 PAT，多負擔）
散佈機制① pip 起手 → 疊 ③最輕、你現在就在雲端 pip；② 容器當備案
strategies/ 歸屬引擎(Repo A)策略定義是共享資產；未來可加 Repo B override 目錄做自訂策略
itemhsu 自己變成使用者 #0dogfooding，親身踩使用者的坑
主要風險：① 這是真錢系統，務必走 P3 平行運轉對帳數日再切。 ② 子策略目錄碼/資料分離要小心 data 路徑。 ③ 遷移期間 schema 跟引擎走，Repo B 資料才有版本依據——golden 回放正是保險。
一句話：分兩 repo 完全可行且正確——切的線是「引擎碼(A，公開、有版本)」對「設定+資料+執行+金鑰(B，私有、每人一份)」。 全程雲端、自託管金鑰不變。唯一硬前提是先做工作目錄解耦（§4）。做完後，分歧問題消失，我們的 compat 測試轉成引擎的發版品質閘。
