✅ P3 上線對帳 Checklist

用薄殼範本站起 Repo B → paper 平行對帳 → 一致才切真錢 · 2026-06-03

引擎 v1.0.0 已就緒金鑰/建 repo 由你親手真錢系統：先 paper、後對帳、再切換
安全原則：所有 API 金鑰、PAT、Secrets 都由你本人在 GitHub 介面或 GUI 輸入， 絕不貼進對話或經我執行。本文件只給「步驟與檢查項」，不含任何金鑰。

步驟 0 — 先決條件

  • 一個 Alpaca paper 帳戶（先用模擬，不碰真錢）。拿到該帳戶的 API Key / Secret。
  • 你的 GitHub 帳號（建 Repo B 用）。
  • 引擎 Release 已存在：itemhsu/tech-rebalance@v1.0.0（已完成）。

步驟 1 — 用範本建 private Repo B

  1. 把引擎 repo 內 examples/repo-b-template/ 的內容複製出來（4 個檔）。
  2. 在 GitHub 建一個 private repo，例如 {你}/my-trading，把這 4 個檔放進去、預設分支 main。
  3. 編輯 accounts.json：填你的帳戶 id / 策略（範本是 top10）/ secret_prefix: ACC1。
⚠️ Repo B 必須 private（裡面有你的資料；金鑰雖在 Secrets 不在檔案，但資料仍屬個人）。

步驟 2 — 產生 ENGINE_INSTALL_TOKEN（讀私有引擎用）

引擎 repo 目前是 private，Repo B 的 workflow 要 pip install 它需要一個能讀取的 token。

  1. GitHub → Settings → Developer settings → Fine-grained personal access tokens → Generate new token。
  2. Resource owner = itemhsu；Repository access = Only select repositories → 勾 tech-rebalance。
  3. Permissions → Repository permissions → Contents: Read-only（只需這一項）。
  4. 有效期設短一點（如 90 天），到期再換。產生後立刻複製。
若你願意把引擎 repo 設為 public（純碼無秘密），這一整步可省略——Repo B 直接公開安裝、免 token。是否公開由你決定。

步驟 3 — 在 Repo B 設 Actions Secrets

Repo B → Settings → Secrets and variables → Actions → New repository secret。逐一加：

Secret 名稱內容必填
ACC1_ALPACA_KEYAlpaca paper API Key✅
ACC1_ALPACA_SECRETAlpaca paper Secret✅
ENGINE_INSTALL_TOKEN步驟 2 的 PAT（引擎 public 則免）視情況
EMAIL_SENDER / EMAIL_PASSWORD / EMAIL_RECIPIENT寄報告（先不設也行）選用
金鑰只貼進 GitHub Secret 輸入框（值不落檔、不入 log）。不要寫進 accounts.json 或任何檔案。

步驟 4 — 跑 dry-run（不下單）

  1. Repo B → Actions → Daily rebalance (thin shell) → Run workflow → dry_run: true。
  2. 看 log：應出現 
    pip 裝 tech-rebalance@v1.0.0 ✅
run-account --account all --dry-run
帳戶 #1 … 計算出持倉/排名（不下單）
  3. 若卡在安裝 → token 權限不足；卡在金鑰 → Secret 名稱拼錯（要 ACC1_ALPACA_KEY）。

步驟 5 — 平行對帳（關鍵關卡）★

讓 Repo B 的 dry-run 與現有系統跑同一天，逐欄比對。兩邊都不下單，純比計算結果。

比對項來源通過標準
當日選股（持倉清單）dry-run log 的 top10 / portfolio symbols完全一致（同一組代號）
市值排名log 的 ranked_stocks 順序名次一致
NAV / cash帳戶狀態同一 paper 帳戶 → 應相同
計畫下單（數量/方向）dry-run 的 orders（SELL/BUY、股數）逐筆一致
報告 data.json schema產出的 data.json通過 data-schema、儀表板能讀
建議對帳天數：連續 3 個交易日都逐欄一致 → 視為通過。涵蓋「每日」與「換股日」更佳。
若有差異：不要切。記錄差異欄位 → 多半是路徑/版本/資料來源問題，回頭查 TR_WORKDIR 與引擎版本。差異消除前維持舊系統運作。

步驟 6 — Go / No-Go + 切換

條件動作
3 天 paper 對帳全一致Go：把 Repo B 的 ACC1_* 換成真錢帳戶金鑰；啟用排程
任一欄不一致No-Go：維持舊系統；查差異

切換與並行

  • 切真錢初期，建議舊系統先停排程但不刪（保留可回退）。
  • Repo B 真錢跑穩數日後，再退役舊路徑（P4）。

步驟 7 — 回退方案（出事就用）

  1. 停用 Repo B 的 daily.yml 排程（Actions → 關閉 workflow）。
  2. 重新啟用舊系統的 daily_all_accounts.yml 排程。
  3. 因為兩邊操作同一個 Alpaca 帳戶，持倉狀態連續，回退不會遺失資料。
我能幫的：對帳時把兩邊 log/data.json 貼給我，我幫你逐欄 diff 找差異； 或在切換前幫你檢查 Repo B 的 accounts.json / workflow 設定。金鑰與建 repo 由你；驗證與除錯我陪你做。
