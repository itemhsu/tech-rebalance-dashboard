🏛 卷商 Schema 分析 6 家整合設計

把現有單一綁定 Alpaca 的架構抽象化，未來 accounts.json 可選擇任一卷商， 每家卷商用一個 JSON 描述其 API。本文先做分析，後續實作分階段進行。

A需求與目標

當前架構限制

  • 整個系統綁死 Alpaca：trader.py 直接呼叫 Alpaca REST 端點
  • accounts.json 只有 alpaca_secret_prefix 欄位，無 broker 概念
  • 切換到其他卷商要改 7+ 個 Python 檔案

目標

  • ✅ 每家卷商有一個 brokers/{id}.json 描述 API（authentication、endpoints、capabilities）
  • ✅ accounts.json 新增 broker 欄位指向卷商 id
  • ✅ 抽象 BrokerClient 介面：6 個共用方法統一所有卷商
  • ✅ 各家 Python 實作為 brokers/alpaca_client.py / brokers/tradier_client.py ...
  • ✅ 策略邏輯 (portfolio.py / momentum_runner.py) 完全不知道背後是哪家卷商

本文件範圍

📋 本份只做分析，不動 production code 產出：(1) 卷商對照 (2) 統一 schema 草案 (3) 各家 JSON 範例 (4) 遷移路線圖。 實際 refactor 在獨立 task 進行。

3TradeStation Tier B · 認證複雜

🏛 TradeStation — 美股 + 選擇權 + 期貨

developer.tradestation.com · 老牌專業券商

認證（OAuth 2.0）

1. Register app at developer.tradestation.com → client_id, client_secret
2. User authorizes → authorization_code (one-time)
3. POST /v2/security/authorize { grant_type: "authorization_code", code }
   → access_token (20 min) + refresh_token
4. Refresh: POST { grant_type: "refresh_token", refresh_token }

Headers:
  Authorization: Bearer {access_token}

Endpoints:
  Sim:   https://sim-api.tradestation.com/v3
  Live:  https://api.tradestation.com/v3

本系統整合注意

  • ⚠️ 需要 OAuth flow + refresh token 管理 — 不能單純存 API key
  • ⚠️ access_token 只有 20 分鐘，必須 refresh logic
  • ❌ 不支援 fractional shares
  • ✅ 期貨支援完整，未來想加入期貨策略可用

5tastytrade Tier A · 選擇權強

🏛 tastytrade — 選擇權 + 期貨 + 美股

developer.tastytrade.com · 選擇權社群知名

認證（兩種方式）

# 方法 A：OAuth 2.0（推薦給 GitHub Actions）
1. Developer Portal 註冊 App → client_id, client_secret
2. 一次性授權取得 access_token + refresh_token
3. 之後每次：用 refresh_token 換新 access_token
4. refresh_token 永不過期（除非用戶撤銷）

# 方法 B：Session login（不建議 unattended 用）
1. POST /sessions { login, password } → session-token (24h)
2. ⚠️ 要存使用者網頁登入密碼，風險高

Endpoints:
  Sandbox: https://api.cert.tastyworks.com
  Live:    https://api.tastyworks.com

本系統整合注意

  • ✅ 用 OAuth refresh_token 模式，跟 GitHub Actions 相容（同 TradeStation）
  • ⚠️ access_token 過期需 refresh，refresh_token 可能 rotate 要更新 GitHub Secret
  • ❌ 絕不要用 session login 把網頁密碼存 GitHub Secret，網頁密碼可控制整個帳戶
  • ❌ 不支援 fractional shares
  • ✅ 官方 Python SDK tastytrade-sdk 維護良好
  • ✅ Notional Market Order（用金額下單，自動算股數）

6永豐金 Shioaji Tier A · 唯一 TW 市場

🏛 永豐金證券 — Shioaji SDK — 台股 + 期貨 + 選擇權

sinotrade.github.io · 永豐唯一官方程式交易 SDK

整合方式（Python SDK，非 REST）

import shioaji as sj
api = sj.Shioaji(simulation=True)  # 模擬 / 正式
api.login(
    api_key    = "YOUR_API_KEY",
    secret_key = "YOUR_SECRET_KEY",
)
api.activate_ca(ca_path="Sinopac.pfx", ca_passwd="...", person_id="A123...")
# 下單需要 CA 憑證（生產環境）

contracts = api.Contracts.Stocks["2330"]  # 台積電
order = api.Order(
    price=0, price_type=sj.constant.StockPriceType.MKT,
    action=sj.constant.Action.Buy, quantity=1,
    order_type=sj.constant.OrderType.ROD,
)
trade = api.place_order(contracts, order)

本系統整合注意

  • ⚠️ 沒有 REST API，必須跑 Python SDK（要 import shioaji）
  • ⚠️ 生產環境下單需金融憑證 (CA, .pfx 檔)，paper 模式不需
  • ⚠️ 持倉/帳戶資料用 callback 推送，不是 polling
  • ❌ 美股不能用此 SDK（永豐 API 只有 TW 商品）
  • ✅ 模擬交易免費，免身分驗證
  • ⚠️ 交易時段：09:00-13:30 TW time（盤後 13:40-14:30 另案處理）
💡 抽象層需處理 Shioaji 是 SDK 模型 vs 其他 5 家是 REST 模型。可能設計兩層 adapter： BrokerClient (interface) → RestBrokerClient (alpaca/tradier/...) + SdkBrokerClient (shioaji)

C統一抽象層

所有卷商必須支援的 6 個共用方法
class BrokerClient(ABC):
    """所有卷商必須實作的最小介面。"""

    def __init__(self, broker_spec: dict, env: dict): ...

    def is_trading_day(self, date: date) -> bool: ...
    def get_account_balance(self) -> AccountBalance: ...
    def get_positions(self) -> list[Position]: ...
    def get_latest_prices(self, symbols: list[str]) -> dict[str, float]: ...
    def place_order(self, symbol: str, qty: float, side: str,
                    order_type: str = "market") -> OrderResult: ...
    def cancel_all_open_orders(self) -> int: ...
    def wait_for_fills(self, order_ids: list[str], timeout: int) -> None: ...

共用資料型別

@dataclass
class AccountBalance:
    nav: float
    cash: float
    buying_power: float
    currency: str = "USD"

@dataclass
class Position:
    symbol: str
    qty: float
    avg_entry_price: float
    current_price: float
    market_value: float

@dataclass
class OrderResult:
    order_id: str
    symbol: str
    side: str           # "buy" | "sell"
    qty: float
    status: str         # "new" | "filled" | "rejected"
    filled_avg_price: float = 0.0
每家實作只需處理自己的 API 細節，外部呼叫者（main.py / portfolio.py）只看 BrokerClient 介面。

H風險與限制

類別風險緩解
抽象層滲漏 各家 API 細節差太多（fractional / TIF / asset class）強塞同一介面會變醜 capabilities 區塊讓策略檢查能力，下單前 validate
SDK vs REST Shioaji 是 SDK callback 模型，整合到 REST polling 風格不自然 分兩層 base class：RestClient + SdkClient
OAuth refresh 機制 TradeStation / tastytrade 用 OAuth 2.0：access_token 過期需 refresh，refresh_token 偶爾 rotate 實作 _ensure_access_token()：過期時自動 refresh，若 refresh_token 變動則用 gh secret set 更新 GitHub Secret。一次設定後永久自動運轉。
市場差異 TW vs US 開盤時間、貨幣、交割天數都不同 strategy schema 也要加 market 欄位（current 預設 US）
歷史資料 Public 沒 bars API；Shioaji 的 bars 跟 US 結構不同 把 market_data 抽出來，可以不從卷商拿（fallback yfinance）
佣金計算 Shioaji TW 有交易稅 / 手續費，跟美股 $0 不同 strategy 或 portfolio.py 加 fee_model 設定
多 broker 部署 GitHub Secrets 數量會暴增（每帳戶多個 secret） 用 prefix 命名規則 + 文件清楚標示
⚠️ 重大風險 若實作有 bug 而誤觸發實單，可能造成損失。 所有新 broker 必須先在 sandbox/paper 跑滿 4 週 + 100+ 次成功訂單後才開放 live。
