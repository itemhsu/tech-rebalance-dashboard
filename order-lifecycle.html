📋 訂單生命週期可觀測性

「交易指令的下落」追蹤 — 設計與實作 · 2026-06-03

trader.pytrade_log.py狀態：核心已實作 · 含跨日對帳

1. 問題

帳號 #3 的 trade_events.jsonl 只有 ORDER_PLANNED（規劃），沒有任何「下落」： 訂單到底有沒有送出？成交了嗎？成交價多少？被拒絕了嗎？無從得知。
  • portfolio.calculate_rebalance 寫 ORDER_PLANNED（意圖）—— dry-run 也會寫（trigger 帶 _dryrun）。
  • trader.execute_rebalance 只寫 ORDER_SUBMITTED，而且 wait_for_fills() 回傳的成交資料被丟棄。
  • record_order_fill / record_order_reject 函式早就存在，卻從沒被呼叫。
  • 休市時下單會排隊、開盤後才成交 → 當下查不到，需隔日對帳，但沒有這段程式。

2. 訂單狀態機

PLANNED ──(live 送出)──▶ SUBMITTED ──┬─(filled)────────▶ FILLED（含成交均價/量） │ ├─(partial)───────▶ FILLED(partial) │ ├─(rejected/expired)▶ REJECTED └─(dry-run)──▶ 不送出、不寫成交事件 └─(休市 pending)──▶ 隔日 reconcile 補記終態

3. 設計

機制時機做什麼
即時追蹤 _record_outcomes送單 + wait_for_fills 之後 逐筆 get_order(oid)：filled→ORDER_FILLED（成交均價/量）、partial→FILLED(partial)、rejected/expired→ORDER_REJECTED、仍 pending→不記終態
跨日對帳 reconcile_outcomes每次 live run 開始時 讀 trade_events.jsonl 找「已 SUBMITTED 但無 FILLED/REJECTED」的 order_id → 查最新狀態 → 補記下落（解決休市排隊、開盤成交）
dry-run 區隔dry-run 不寫任何成交事件；ORDER_PLANNED 以 trigger _dryrun 標示，log viewer 應淡化顯示「規劃（未送出）」

4. 實作（已完成）

# trader.execute_rebalance（live 路徑）
reconcile_outcomes(client, account_id, strategy)   # ① 開始：補記昨日 pending
... 送 SELL / BUY，記 ORDER_SUBMITTED（含 order_id）...
_record_outcomes(client, submitted, ...)           # ② 結束：記今日 FILLED/REJECTED/PARTIAL

# trade_log 新增 read_events() 供對帳去重（已 FILLED/REJECTED 的 order_id 不重記）
結果：每個 order_id 形成完整鏈 PLANNED → SUBMITTED → FILLED/REJECTED， 含成交均價、成交量、最終狀態。dry-run 不再混入成交事件、且明確標示。

5. 測試（7 案例，全綠）

測試驗證
_record_outcomes filled / rejected / partial / pending各狀態記對的事件；pending 不記終態
reconcile 補記 fill / reject昨日 SUBMITTED 未結 → 查到成交/拒絕 → 補記
reconcile 去重已有 FILLED 的 order_id 不重複記

6. 後續（建議）

  • 報告/dashboard「訂單下落」視圖：把 PLANNED→SUBMITTED→FILLED 串成一列顯示成交價與滑價。
  • log viewer 淡化 dry-run：trigger 含 _dryrun 的 PLANNED 以灰色標「規劃（未送出）」。
  • 未成交告警：對帳發現超過 N 天仍未結的 SUBMITTED → 開 issue / email 告警。
