B根因分析

為什麼會這樣？

追溯 commit 史，每個重複都源於一個合理但短視的選擇：

  1. 新增策略時急著上線 → 複製 main.py 改成 d2p2t6/main.py 最快
  2. 沒人重構 → 第 3 個策略時 universal runner 已該寫，但又複製了一份
  3. backtest 那邊有 SpecEngine 統一了 → production 卻沒同步遷移

對比：backtest 已做對的部分

面向backtest (✅)production (❌)
引擎1 個 SpecEngine 跑 14 策略5+ 個 main.py
新增策略寫一個 v3 JSON寫 JSON + 改 5+ Python
選股邏輯JSON-driven 統一分散在每個 runner
單元測試28 個 metrics + 13 schema各 runner 各自寫
結論：把 backtest 的 SpecEngine 設計搬到 production 即可解決大半問題。 再加 BrokerClient（已完成）就能徹底切離卷商與策略邏輯。

1ACCOUNT_STRATEGY 硬編 P0 · 快修

⏱️ 30 分鐘 · 風險極低 · 修立刻見效的 bug

問題

scripts/send_email_from_data.py
ACCOUNT_STRATEGY = {
    "1": "top10",
    "2": "d2p2t6",
    "3": "top9psq",      # ← 帳戶 #3 現在是 mom_6m_t20，這裡還寫 top9psq
}                         # ← 帳戶 #4 / #5 完全不在表內 → 漏寄信

解法

def _account_strategy(account_id: str) -> str | None:
    accounts = json.loads(ROOT / "accounts.json").read_text())["accounts"]
    for acc in accounts:
        if str(acc["id"]) == str(account_id):
            return acc.get("strategy")
    return None

影響

  • ✅ 帳戶 #3-5 自動寄信
  • ✅ 未來新帳戶不用碰程式碼

3Universal Runner P1 · 關鍵

⏱️ 3 天 · 取代 5 個 main.py / 2 個 runner.py

新架構

runner.py（新檔，~350 行）

def run(account_id: str, strategy_id: str, data_dir: str, dry_run: bool):
    # Step 1: build client (BrokerClient 已抽象)
    client = build_client_for_account(account_id)

    # Step 2: 守門（交易日 + frequency）
    if not _gate(client, strategy_spec, dry_run):
        return  # skip + trade_log REBALANCE_SKIP

    # Step 3: 選股（statisfies all 5 selection methods）
    picks = select_portfolio(strategy_spec, client)

    # Step 4: 算 rebalance（已通用：pf.calculate_rebalance）
    nav, cash = client.get_account_nav()
    positions = client.get_current_positions()
    orders = pf.calculate_rebalance(
        positions, picks, prices, nav, cash,
        trigger=_trigger_code(strategy_spec),
        account_id=account_id, strategy=strategy_id,
    )

    # Step 5: 執行（已通用：tr.execute_rebalance）
    tr.execute_rebalance(client, orders, dry_run,
                         account_id=account_id, strategy=strategy_id)

    # Step 6: state + history + harvest（全部統一）
    _post_run(client, orders, data_dir, strategy_spec)

選股統一：selection.py

def select_portfolio(spec: dict, client: BrokerClient,
                     today: date = None) -> list[str]:
    """對應 backtest SpecEngine._select_at 的 live 版本。"""
    method = spec["selection"]["method"]

    if method == "top_n_by_metric":        return _sel_top_n_by_metric(spec, client)
    if method == "top_n_per_group":        return _sel_top_n_per_group(spec, client)
    if method == "factor_score":           return _sel_factor_score(spec, client)
    if method == "weighted_percentile":    return _sel_weighted_percentile(spec, client)
    if method == "weighted_percentile_per_group":
                                            return _sel_wp_per_group(spec, client)
    raise ValueError(...)

跟 backtest 共用嗎？

分離但對齊：backtest 用 yfinance/parquet，live 用 broker API。 介面 (spec → list[str]) 完全相同，邏輯實作平行寫但可寫共用 helpers （如 _score_factors()）。

解法：JSON spec 內 universe 區塊已能描述

  • grouped — 多組（d2p2t6 用）
  • exchange_filter — 條件過濾（toprank）

新模組 universe_loader.py（live 版本）

def load_universe(spec: dict, client: BrokerClient) -> dict[str, list[str]]:
    """讀 spec.universe 回傳 {group_id: [symbols]}。

    對應 backtest 的 SpecEngine._resolve_universe，但 prices 用 broker API。
    """
    u = spec["universe"]
    if u["type"] == "single":
        return {"__all__": _load_source(u["source"])}
    if u["type"] == "grouped":
        return {g["id"]: _load_source(g["source"]) for g in u["groups"]}
    ...

動態 market_cap 排名

各 universe.py 都在做「市值排名」 — 但其實 BrokerClient.get_latest_prices() + 靜態 shares_outstanding.json 即可。把這個邏輯抽到 selection.py 即可。
